# ToothGroupNetwork — three DISTINCT variants

_Area key: `ToothGroupNetwork`_

## Purpose
Point-cloud tooth INSTANCE + semantic (FDI-numbered) segmentation of 3D intraoral scan meshes (.obj/.stl/.ply). All three folders descend from the MICCAI-2022 3D Teeth Scan Segmentation challenge-winning "ToothGroupNetwork" (Team CGIP), but they are genuinely different variants, not copies. Head-to-head: (1) "ToothGroupNetwork-main 3" (Oct 2023) is the canonical/original upstream repo — module-namespaced models, plain torch.load, six models offered, but NO weights on disk (README links a Google-Drive ckpts.zip). (2) "ToothGroupNetwork-main" (Aug 2024) is a personal clinical FORK of variant 3: its inference code (inference_pipeline_maker.py, start_inference.py, gen_utils.py) is byte-identical to variant 3, but it is the ONLY variant that ships trained weights on disk (8 .h5 files, ~250MB, which are actually PyTorch zip checkpoints despite the .h5 extension), plus an unrelated MRI/ U-Net NIfTI experiment, Docker/shell helper scripts, and a personal "Commands for my specific use case.txt". (3) "ToothGroupNetwork-main 2" (Sep 2024) is the heaviest REFACTOR — rebranded "Teeth Segmentation Models", adds a Plotly/Dash web app, a Dockerfile + gdown dataset downloader, flattened top-level model classes, an EXTRA PointMLP model, a generalized load_checkpoint (.pth/.h5/.safetensors), and a pipeline that takes a mesh object + PCA-align flag instead of a file path — but also has NO weights on disk. The actual neural inference is NOT web-portable: every variant depends on hand-written CUDA extensions (external_libs/pointops, pointnet2_utils) for furthest-point-sampling / kNN / grouping and calls .cuda() unconditionally. Best porting SOURCE for a web app: variant 2 for its cleaner, decoupled mesh-object pipeline, safetensors support, and existing browser-facing UI patterns (FDI label map, Dash callbacks); variant A is the best WEIGHTS source (only one with checkpoints on disk). For CBCTer, which already has a client-side ONNX tooth-segmentation pipeline, none of these run in-browser as-is; the reusable value is the geometry/preprocessing math (normalization, FPS resampling, boundary-aware sampling, KDTree label propagation, colored-mesh + FDI mapping), which CBCTer could re-implement in JS/onnxruntime-web.

**Tech stack:** Python 3.8-3.11, PyTorch (1.7.1 in variant A/3 Docker; 2.3-2.4 in variant 2), CUDA (mandatory — all pipelines call .cuda(); RTX40xx issues noted), Custom CUDA C++ extensions: external_libs/pointops (FPS/kNN/grouping/aggregation/subtraction/interpolation) and external_libs/pointnet2_utils — NOT web-portable, Open3D (mesh/point-cloud IO, normals, subdivision, visualization), trimesh (OBJ/STL/PLY loading, used in web upload path), NumPy / scikit-learn (KDTree, PCA, KMeans clustering), CBL Point Transformer backbone (models/modules/cbl_point_transformer), Dash + Plotly (web UI — variant 2 web_app.py only), gdown (dataset/checkpoint download — variant 2), safetensors (variant 2 load_checkpoint only), wandb (training experiment tracking), nibabel + torchvision + OpenCV (variant A MRI/ NIfTI U-Net experiment only — unrelated to tooth pipeline), Docker (CUDA images: pytorch:1.7.1-cuda11.0 in A/3 README, pytorch:2.3.1-cuda11.8 in variant 2 Dockerfile)

**License:** No LICENSE file present in any of the three variant folders; license is unstated. Upstream is limhoyeon/ToothGroupNetwork (MICCAI-2022 challenge winner). Reference codebases cited in README (contrastBoundary, Pointnet_Pointnet2_pytorch, POSTECH point-transformer, fxia22/pointnet.pytorch, WangYueFt/dgcnn) carry their own licenses. Treat as research-only / license-unknown.

**I/O formats:** INPUT: 3D intraoral scan meshes — .obj (primary, named {ID}_lower.obj / {ID}_upper.obj), .stl, .ply (variant 2 web upload + SmartRPD STL path), INPUT meshes must be pre-oriented (axis-aligned; Y-axis to back, same Z for upper/lower); variant 2 can auto-align via PCA flag, OUTPUT: per-vertex labels as challenge-format JSON {id_patient, jaw, labels (FDI 11-48), instances} — same schema as GT json files, INTERMEDIATE: preprocessed farthest-point-sampled vertices saved as .npy (preprocess_data.py), TRAINING GT: per-vertex .json label files (data_json_parent_directory) or .txt label lists (SmartRPD), WEIGHTS: PyTorch checkpoints — .h5 files (variant A, actually torch zip archives), also .pt/.pth/.safetensors accepted by variant 2 load_checkpoint, UNRELATED (variant A MRI/ only): NIfTI volumes via nibabel, .pt tensor dumps

## Modules

### ToothGroupNetwork-main
Personal CLINICAL FORK of the original repo (variant 3), tailored for a CUDA Docker workflow (README walks through a pytorch:1.7.1-cuda11.0 image and Osaka-U/UCEAP test cases). Its core inference code is BYTE-IDENTICAL to variant 3 (inference_pipeline_maker.py, start_inference.py, gen_utils.py all diff-clean against variant 3; only train_config_maker.py differs in the wandb entity/on flag). What makes it unique and high-value: it is the ONLY variant that ACTUALLY SHIPS TRAINED WEIGHTS on disk — models/*.h5 for tgnet_fps (64MB), tgnet_bdl (0.5MB), tsegnet (9.7MB), tsegnet_centroid (3.4MB), pointnet (37MB), pointnetpp (36MB), pointtransformer (32MB), dgcnn (4MB). NOTE: despite the .h5 extension these are PyTorch checkpoint zip archives loaded via torch.load (confirmed: `file` reports 'Zip archive data'), NOT Keras/HDF5. It also bundles a SEPARATE, UNRELATED experiment in MRI/Try1/ (a 2D/3D brain-segmentation U-Net loaded from torch.hub 'mateuszbuda/brain-segmentation-pytorch', reading NIfTI via nibabel, GeneralizedDiceLoss) that has nothing to do with the tooth point-cloud pipeline. Adds personal automation: render.sh/batch_process.sh/transfer.sh/helper.sh/convert.sh/postProcess.sh, convert.py, test_preprocessor.py (data re-orientation), test_data_setup.py, generator/runner/trainer .py, and 'Commands for my specific use case.txt'. Inference: tgnet two-stage (FPS GroupingNetworkModule -> boundary-aware GroupingNetworkModule), driven by start_inference.py -> ScanSegmentation.process -> writes challenge-format JSON {jaw,labels,instances}. Tailored as a ready-to-run, weights-included clinical batch inference setup inside Docker.

- **Algorithm:** Tooth Group Network (tgnet): two-stage point-cloud grouping net — stage1 farthest-point-sampling GroupingNetworkModule (CBL point-transformer backbone) predicts semantics + per-point offset, clustered into instances; stage2 boundary-aware-sampled GroupingNetworkModule refines tooth boundaries; KDTree propagates labels back to full mesh; PCA + center-line heuristic assigns FDI quadrant numbers. Also bundles TSegNet, PointNet, PointNet++, DGCNN, PointTransformer. CUDA pointops (FPS/kNN/grouping) required.
- **Entry points:** `start_inference.py`, `inference_pipelines/inference_pipeline_maker.py`, `inference_pipelines/inference_pipeline_tgn.py`, `predict_utils.py`, `gen_utils.py`, `ops_utils.py`, `models/modules/grouping_network_module.py`, `models/*.h5 (trained weights, present on disk)`, `MRI/Try1/mriSegmentationModel.py (separate NIfTI U-Net experiment)`

### ToothGroupNetwork-main 2
Heaviest REFACTOR of the three, rebranded 'Teeth Segmentation Models'. Most modernized and most web-oriented variant, but ships NO weights on disk (expects ckpts downloaded from Google Drive). Distinct additions: (a) web_app.py — a Plotly/Dash browser UI: drag-and-drop OBJ/STL upload via trimesh, render as go.Mesh3d, 'Label Teeth' button runs the pipeline, optional PCA-alignment toggle, FDI_NUMBERING dict (11-48 -> human tooth names) for hover labels; (b) Dockerfile (pytorch:2.3.1-cuda11.8, launches JupyterLab) + requirements.txt (adds spconv-cu120, torch_scatter, timm, transformers, accelerate, dash) + environment.yml (python 3.11, torch 2.4, includes safetensors); (c) download_miccai22.py — gdown script pulling the challenge OBJ meshes + GT JSON + splits; (d) metrics.py and eval_results.py for batch evaluation; (e) preprocess_data_smartrpd.py — a SEPARATE preprocessing path for STL 'SmartRPD' data (PCA-aligns vertices, resamples gums/teeth to 12000 pts each). Architectural refactor vs variant 3: models are FLATTENED to top-level classes (models/pointnet.py PointNet, models/dgcnn.py DGCnn, models/point_transformer.py PointTransformer, models/tgn.py GroupingNetworkModule) instantiated with NO config dict (hyperparams hardcoded in __init__); ADDS models/pointmlp.py (PointMLP, ~17KB) — a model NOT present in variants A/3; DROPS tsegnet from the inference model menu though tsegnet_model.py still exists. make_inference_pipeline signature changed to (model_name, ckpt_path, bdl_ckpt_path=None) and uses gu.load_checkpoint (supports .pt/.pth/.h5 AND .safetensors) instead of raw torch.load; the tgn pipeline __call__ now accepts a MESH OBJECT or path plus a pca flag (variant A/3 accept only a path), and uses torch.load(..., weights_only=True). Best-suited porting source for UI/pipeline shape, FDI mapping, and safetensors weight format.

- **Algorithm:** Same tgnet two-stage FPS + boundary-aware grouping core as upstream, but exposed through a decoupled mesh-in/dict-out pipeline with optional PCA pre-alignment; adds PointMLP as an extra single-stage point classifier. Still CUDA-bound via external_libs/pointops; web_app.py is only a front-end wrapper around the same GPU pipeline.
- **Entry points:** `web_app.py (Plotly/Dash UI)`, `start_inference.py`, `inference_pipelines/inference_pipeline_maker.py`, `inference_pipelines/inference_pipeline_tgn.py`, `models/tgn.py`, `models/pointmlp.py (unique to this variant)`, `models/pointnet.py / dgcnn.py / point_transformer.py (flattened classes)`, `gen_utils.py (load_checkpoint with safetensors)`, `download_miccai22.py`, `preprocess_data_smartrpd.py`, `Dockerfile`, `metrics.py`, `eval_results.py`

### ToothGroupNetwork-main 3
The CANONICAL ORIGINAL upstream repo (Team CGIP, Ho Yeon Lim & Min Chang Kim) — README explicitly states this is the MICCAI-2022 3D Teeth Scan Segmentation challenge first-place code (arXiv:2305.18277), dated Oct 2023, the earliest of the three. It is the baseline that variants A and 2 both descend from. Offers six models (tsegnet | tgnet | pointnet | pointnetpp | dgcnn | pointtransformer) with the ORIGINAL module-namespaced layout: models/modules/{grouping_network_module,tsegnet,point_transformer,dgcnn,pointnet,pointnet_pp}.py, instantiated WITH a config/model_info dict; make_inference_pipeline takes (model_name, ckpt_path_ls) and loads via plain torch.load(ckpt_path_ls[i]) — no safetensors, no load_checkpoint helper, no pca flag, pipeline __call__ takes a file path only. Ships NO weights on disk (README directs users to Google-Drive ckpts(new).zip; also references a challenge_branch for the exact challenge checkpoints). No web app, no Docker, no MRI experiment, no PointMLP. Contains the full TRAINING toolchain (start_train.py, trainer.py, preprocess_data.py, train_configs/) and the most thorough README documenting training of tgnet_fps then tgnet_bdl, and tsegnet's two-stage centroid->segmentation training. Has a stray `from this import d` line in base_model.py that the later variants removed. Best-suited as the reference for ALGORITHM SEMANTICS and training, but not for porting (no weights, oldest API).

- **Algorithm:** Reference implementation of the challenge-winning tgnet (two-stage farthest-point + boundary-aware-sampling grouping network with per-point offset prediction and clustering) plus TSegNet (centroid prediction module + tooth segmentation module) and direct point-cloud baselines (PointNet/PointNet++/DGCNN/PointTransformer). All depend on CUDA external_libs/pointops and pointnet2_utils.
- **Entry points:** `start_inference.py`, `start_train.py`, `trainer.py`, `preprocess_data.py`, `inference_pipelines/inference_pipeline_maker.py`, `inference_pipelines/inference_pipeline_tgn.py`, `inference_pipelines/inference_pipeline_tsegnet.py`, `models/modules/grouping_network_module.py`, `models/modules/tsegnet.py`, `train_configs/train_config_maker.py`, `gen_utils.py`, `ops_utils.py`

## Reusable utilities

- `gen_utils.py (all variants; v3/A identical, v2 adds load_checkpoint)` — Core self-contained mesh/point-cloud helpers, the highest-value port candidates: read_txt_obj_ls (OBJ parse to vertices+normals), load_mesh (Open3D load + clean), resample_pcd (drop/duplicate to N points, uniform or FPS), axis_rotation (Rodrigues rotation matrix — pure numpy, web-portable), count_unique_by_row (vectorized per-row label voting for boundary sampling), get_colored_mesh / np_to_pcd_with_label (FDI palette coloring), load/save_json, sigmoid. NOTE fps() itself calls CUDA pointops — would need a JS FPS replacement. v2 adds load_checkpoint (.pt/.pth/.h5/.safetensors dispatch).
- `inference_pipelines/inference_pipeline_tgn.py (all variants)` — The end-to-end algorithm worth re-implementing: vertex centering + min-max normalization (scaler=1.8, shifter=0.8), midpoint subdivision when <24000 verts, 24000-pt FPS sampling, boundary-aware resampling (get_boundary_sampled_feats: KDTree k=40 neighborhood label-ratio test), KDTree k=1 label propagation back to full mesh, PCA center-line heuristic mapping instance clusters to FDI quadrant numbers (the +8/+10/+20 jaw offset math). All pure numpy/sklearn/Open3D except the two model() GPU calls.
- `predict_utils.py (all variants, identical)` — ScanSegmentation wrapper: jaw detection from filename or OBJ header, NpEncoder for numpy->JSON, write challenge-format output {id_patient,jaw,labels,instances}, jaw-based label offset (lower += 20). Fully self-contained, framework-agnostic, directly portable.
- `ops_utils.py (all variants)` — Clustering utilities used by the instance stage: get_clustering_labels, clustering_points (kmeans/other) over predicted foreground offset points. Mostly numpy/sklearn; reusable port candidate for turning offsets into tooth instances.
- `preprocess_data_smartrpd.py (variant 2 only)` — Standalone STL preprocessing: PCA-align vertices (flip Y), read .txt per-vertex labels with FDI remap, resample gums/teeth to 12000 pts each, save .npy. Self-contained example of a non-OBJ ingest path.
- `download_miccai22.py (variant 2 only)` — Self-contained gdown downloader for the MICCAI-2022 challenge OBJ meshes + GT JSON + train/val/test split txts; useful as a data-fetch reference.
- `web_app.py FDI_NUMBERING dict + label_map (variant 2 only)` — FDI tooth-number -> human-readable name mapping (11..48) and the Dash/Plotly mesh-rendering + upload-and-label callback pattern — directly relevant as a UI reference for a web port.

## Web-portability assessment

**Overall:** ⛔ desktop-only

- **Baseline point-cloud models (TSegNet, PointNet, PointNet++, DGCNN, PointTransformer, PointMLP)** — Alternative single/two-stage point-cloud classifiers bundled alongside tgnet; PointMLP is unique to variant 2. _(depends on: Neural point-cloud tooth inference)_ These are research baselines for the challenge paper, not clinical deliverables. - **Mesh normalization + 24000-pt FPS resampling preprocessing (inference_pipeline_tgn front half)** — Vertex centering + min-max normalization (scaler=1.8, shifter=0.8), midpoint subdivision when <24000 verts, then 24000-point farthest-point-sampling — the pure-geometry preprocessing that feeds the net. _(depends on: Intraoral mesh import (OBJ/PLY) path)_ Only useful if CBCTer adds surface-scan support OR if a server-side model is adopted and you want client-side preprocessing to shrink upload size. - **Boundary-aware resampling (get_boundary_sampled_feats: KDTree k=40 label-ratio test)** — Stage-2 sampler: for each point, KDTree k=40 neighborhood, vote label ratios (count_unique_by_row) to detect tooth boundaries and resample density there. _(depends on: Neural point-cloud tooth inference, Mesh normalization + FPS preprocessing)_ Tightly coupled to the two-stage neural pipeline — only worth porting if the full TGN inference is brought in (server or otherwise). Low standalone value.
- **KDTree k=1 label propagation back to full mesh** — After inference on the subsampled point set, propagate per-point labels to every original vertex via nearest-neighbor (KDTree k=1). _(depends on: Mesh normalization + FPS preprocessing)_ Reusable beyond TGN: any sparse-to-dense label/scalar transfer on a mesh (e.g. mapping a decimated-mesh result back to full-res). Good low-effort building block.
- **Offset-clustering instance extraction (ops_utils get_clustering_labels / clustering_points)** — Turns predicted per-point foreground offsets into tooth instances via kmeans/clustering over shifted points. _(depends on: Neural point-cloud tooth inference)_ Conceptually redundant with CBCTer's existing watershed instance splitting, just in a different geometric domain. Skip unless going full point-cloud.
- **FDI quadrant numbering heuristic (PCA center-line + jaw offset) and FDI_NUMBERING map** — Maps instance clusters to FDI tooth numbers 11-48 via PCA principal axis + center-line midline split and the +8/+10/+20 lower-jaw offset math; plus variant 2's FDI->human-name dict for labels/hover. Highest-value conceptual import. The PCA + center-line logic transfers to centroids of CBCTer's voxel instances with minor adaptation.
- **Challenge-format JSON output writer (predict_utils ScanSegmentation / NpEncoder / jaw label offset)** — Self-contained, framework-agnostic wrapper: detect jaw from filename/OBJ header, encode numpy->JSON, write {id_patient, jaw, labels, instances} with lower-jaw label offset (+20). _(depends on: FDI quadrant numbering heuristic)_ Pure, dependency-free, directly portable. Value is interoperability (matching a recognized challenge result format) rather than new user-facing capability.
- **Intraoral mesh import (OBJ/PLY load + clean + PCA auto-alignment)** — read_txt_obj_ls / load_mesh (Open3D/trimesh OBJ-STL-PLY load, normals, clean) + variant 2's PCA-align flag and SmartRPD STL ingest (PCA-align vertices, flip Y, resample gums/teeth). Enabler for everything else TGN-related, but only worth it if CBCTer wants to handle intraoral scans at all — a product-scope decision. Independently, OBJ/PLY import is a reasonable feature for a dental viewer. axis_rotation (Rodrigues) and the PCA-align routine are clean standalone TS utilities (re-orient any imported mesh to a canonical frame).
- **MICCAI-22 dataset/checkpoint downloaders (download_miccai22.py gdown, variant 2)** — gdown scripts pulling challenge OBJ meshes + GT JSON + split txts (and the Google-Drive ckpts.zip referenced by all variants). Only relevant if someone trains/evaluates models offline. gdown + Google-Drive sourcing is brittle; note CBCTer memory already flags that large model/sample binaries dehydrate from the downloads folder.

## Already covered by CBCTer (skip)

- Tooth segmentation as a clinical outcome — CBCTer already ships a client-side tooth-segmentation pipeline (MONAI UNet via onnxruntime-web in a worker), watershed instance splitting, in-browser tooth-library generation, and an STL/arch review UI. Note the domains differ: CBCTer segments VOLUMETRIC CBCT voxels; TGN segments INTRAORAL SURFACE-MESH point clouds. The end product overlaps even though the input modality does not.
- Mask-to-mesh surface extraction and STL/PLY export — src/lib/segmentation/maskMesh.ts + src/lib/surface/generateSurface.ts already do voxel-surface and marching-tetrahedra meshing with area/volume metrics. TGN's Open3D mesh export adds nothing here.
- Tooth-instance separation from a segmentation — CBCTer's watershed.ts + connectedComponents.ts already turn a foreground blob into per-tooth instances; TGN's offset-clustering (ops_utils clustering_points) is a parallel approach, not a missing capability.
